### Нефункциональные требования

| № | Требование |
|---|---|
| 1 | **Отказоустойчивость**: Система должна обеспечивать доступность 99.95% |
| 2 | **Расширяемость**: Система должна быть расширяемой |
| 3 | **Время реакции на инциденты**: От обнаружения до уведомления оператора не более 5 секунд |
| 4 | **Аналитика в реальном времени**: Система видеоаналитики должна реагировать в реальном времени (миллисекунды) |
| 5 | **Синхронизация**: Задержка синхронизации между edge и центром не должна быть длинной |

### Решение

#### Основное решение: Гибридная архитектура с максимальным переиспользованием существующей инфраструктуры

**Описание**: Система строится как расширение существующей платформы АгроТех с добавлением специализированных компонентов для свиноводства. 

**Ключевые компоненты**:
- Единая система мониторинга, интегрированная в существующую инфраструктуру
- Использование Apache Kafka как центральной шины данных
- Переиспользование существующих хранилищ (ClickHouse, MinIO)
- Расширение текущих мобильного приложения и веб-портала
- Edge-компоненты на фермах для автономной работы

**Преимущества**:
- Минимальные затраты на разработку базовой инфраструктуры
- Единая точка управления всеми системами АгроТех
- Использование накопленной экспертизы команды
- Быстрый запуск MVP за счет готовых компонентов

**Логика принятия решения**:
Данный подход позволяет максимально быстро запустить MVP, используя проверенные технологии и существующую инфраструктуру. Команда уже знакома с технологическим стеком, что снижает риски и время разработки.

### Альтернативы

#### Альтернативное решение: Микросервисная архитектура с разделением Edge и Cloud

**Описание**: Система строится как независимое решение с четким разделением на Edge (на фермах) и Cloud (централизованная) части.

**Ключевые компоненты**:
- Автономные Edge-системы на каждой ферме
- Отдельная облачная платформа для свиноводства
- Новое специализированное мобильное приложение
- Использование облачных ML-сервисов
- Интеграция с корпоративными системами через API

**Преимущества**:
- Полная автономность решения
- Возможность легкой продажи как SaaS
- Оптимизация под специфику свиноводства
- Независимость от корпоративной инфраструктуры

**Недостатки**:
- Дублирование функциональности
- Увеличение сроков разработки
- Необходимость поддержки двух независимых систем
- Сложность интеграции с существующими процессами

### Недостатки, ограничения, риски

**Для основного решения**:

**Недостатки**:
- Зависимость от существующей инфраструктуры может ограничить гибкость
- Необходимость следовать корпоративным стандартам может замедлить инновации
- Сложность выделения в отдельный SaaS-продукт в будущем

**Ограничения**:
- Производительность ограничена пропускной способностью Kafka
- Необходимость обеспечения совместимости с legacy-системами
- Ограничения существующей системы прав доступа

**Риски**:
- Перегрузка центральной Kafka при масштабировании
- Конфликты версий при обновлении общих компонентов
- Сложность изоляции данных разных направлений бизнеса
- Зависимость от стабильности корпоративной инфраструктуры

**Митигация рисков**:
- Использование отдельных топиков Kafka для свиноводства
- Версионирование API для обратной совместимости
- Резервирование критических компонентов на Edge
- Мониторинг производительности и проактивное масштабирование
